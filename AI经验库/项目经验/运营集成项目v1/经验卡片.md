# 运营集成项目v1 - 经验卡片

更新时间：2026-06-16

## 经验 1：大表格画像不要用会触发深递归或大对象展开的写法

### 背景

Excel 上传解析在 `profileTable()` 处理大表格时出现 `RangeError: Maximum call stack size exceeded`。临时限制 5000 行只能绕过问题，不能作为长期方案。

### 结论

表格画像应该采用渐进采样、分块统计和上限保护，避免在大数组、大对象、嵌套结构上使用容易触发调用栈或内存峰值的模式。

### 可复用做法

- 明确最大分析行数、最大列数、最大样本数。
- 分块累积统计，不把完整大表复制到多个中间结构。
- 宽表日序列提取只扫描必要列，并设置日期列候选上限。
- 前端提示“已抽样分析”，不要假装全量画像。
- 后续若继续放大数据量，优先迁移到 Web Worker 或服务端异步任务。

## 经验 2：Chrome 翻译会改 DOM，图表文本区域要主动隔离

### 背景

Chrome 翻译插件可能改写 Recharts 的 `Text` 节点，页面切换时 React 执行 `removeChild` 报错。

### 结论

图表、SVG 文本、动态渲染密集区域应加 `notranslate`，并避免让翻译插件介入虚拟 DOM 管理的节点。

### 可复用做法

- 在应用根节点或图表容器添加 `className="notranslate"`。
- 必要时设置 `translate="no"`。
- 对 Recharts 这类第三方图表组件，优先减少可被插件改写的动态文本节点。
- 测试时至少覆盖“打开 Chrome 翻译插件后切换页面”。

## 经验 3：Agent 执行成功不等于用户看到了结果

### 背景

`/api/gemini/run-workflow` 返回 200，服务端日志也显示 DeepSeek 调用成功，但页面跳回“全平台中控数据看板”，用户看不到本次 Agent 输出，体验上像失败。

### 结论

AI 工作流的完成闭环必须包含：请求成功、视图保持、结果展示、空结果提示、失败提示和日志更新。日志刷新不应该改变主视图。

### 可复用做法

- 执行按钮使用 `type="button"`。
- 成功后显式保持当前业务视图。
- 将 `jsonResult.result` 写入稳定的结果状态。
- 200 但结果为空时给用户提示。
- 接口失败时在当前面板展示错误，不跳页。
- `fetchWorkflowLogs()` 只更新日志，不修改 `activeTab`。

## 经验 4：生产启动脚本必须确认服务的是构建产物

### 背景

生产模式下当前 3001 进程应是 `node dist/server.cjs`，不是 `tsx server.ts`。如果生产启动仍返回 Vite dev HTML，会导致浏览器请求 `/@vite/client` 失败。

### 结论

前后端混合项目需要明确区分开发模式和生产模式，并在验证时检查首页资源路径。

### 可复用做法

- `npm run build` 后用 `npm run start` 启动。
- 检查首页是否加载 `/assets/index-*.js`。
- 确认不加载 `/@vite/client`。
- 保留 `/api/health` 做最小服务端探针。
