## 经验：阿里云部署——更新后页面还是旧的

### 场景
`git pull` + `npm run build` + `pm2 restart` 之后，浏览器访问到的仍是旧版本。PM2 显示 `online` 但实际服务不是最新。

### 结论
服务器上存在脱离 PM2 控制的**僵尸进程**绑定端口。PM2 启停管不到它，`pm2 restart` 对端口占用毫无作用。本质是同一个端口被两个不同路径的 Node 进程竞争。

### 步骤
1. `lsof -i :3000` 看端口归属
2. 找到不属于当前 PM2 进程的 PID（旧部署路径的进程）
3. `kill [PID]` 杀掉僵尸
4. `pm2 restart omni-flow` 重启
5. 浏览器 Ctrl+F5 强制刷新（清缓存）

### 坑点
- PM2 `status: online` 不代表服务正常——新进程可能拿不到端口
- 旧部署目录（如 `/opt/omniflow/`）残留进程不会自动清理
- 浏览器缓存也会导致看到旧页面（F12 → 清空缓存硬性加载）
- `pm2 restart` 只重启同路径进程，管不到不同路径的旧进程

### 下次复用方式
遇到"部署后还是旧的"→ 先 `lsof -i :3000`，再 `ps aux | grep omniflow` 找僵尸

### 状态
已验证 ✅

## 经验：PM2 启动 npm 项目的正确命令

### 场景
部署 OmniFlow 时，试了 `pm2 start dist/server.cjs --name omni-flow` 导致路径锁定，试了 `pm2 start npm -- run start:prod` 报错 script not found。

### 结论
正确命令：
```bash
pm2 start npm --name "omni-flow" -- run start
```
- 项目 package.json 中只有 `"start": "node dist/server.cjs"`，没有 `start:prod`
- 变更部署目录时必须 `pm2 delete` + 重新 `pm2 start`，不能只 `restart`

### 坑点
- `pm2 start dist/server.cjs` 会锁定绝对路径，后续改目录后无法生效
- 名字用 `omni-flow`（带连字符），不是 `omniFlow`

### 状态
已验证 ✅

## 经验：端口冲突 —— 默认 3001 被占

### 场景
OmniFlow server.ts 默认端口 `3001`，阿里云服务器上已有测试项目用 3001 端口。

### 结论
- 在 `.env` 中写 `PORT=3000` 覆盖默认值
- 阿里云安全组需手动放行新端口（入方向 TCP，端口 3000，来源 0.0.0.0/0）

### 状态
已验证 ✅
